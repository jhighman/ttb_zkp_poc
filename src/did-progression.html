Implementation Progression

This page demonstrates the step-by-step progression for implementing Decentralized Identifiers (DIDs) and Verifiable Credentials in our Zero-Knowledge Proof job application system. Each step builds upon the previous one, allowing for incremental adoption and value delivery.

The implementation follows the W3C standards for DIDs and Verifiable Credentials, ensuring interoperability with other systems and future-proofing the application.

The percentage indicators (20%, 40%, 60%, 80%, 100%) represent the conceptual progression through the DID implementation steps in the application, not actual implementation progress. They illustrate the logical sequence of building a complete DID-based identity system.

üîí Security & Trust:

All credentials, proofs, and job postings in this system include expiration dates and status mechanisms to ensure security and maintain trust. Expiry prevents misuse of stale data, reinforces attribution to trusted sources (Trua Trust Bureau or employers), and aligns with W3C standards. The system supports real-time revocation checks without exposing sensitive data, further enhancing privacy and security.

Step 1

DID Implementation

Start 20% Complete Complete

The first step is to implement Decentralized Identifiers (DIDs) for both employers and applicants. DIDs provide a way to identify entities without relying on a centralized authority, enabling self-sovereign identity.

Implementation Details

  • Add DID fields to Applicant and Employer models
  • Implement DID generation and validation
  • Create endpoints for DID resolution
  • Update the UI to display DIDs

Current Status

‚úÖ Implemented

DID support has been added to the backend models and controllers. The system now supports did:web and did:key methods, with proper validation and resolution.

Step 2

Job Listing Verification through Employer DIDs

Start 40% Complete Complete

The second step is to establish verifiable job listings through employer DIDs. This ensures that job postings are legitimately offered by the companies they claim to represent, eliminating the need for trusted intermediaries and ensuring applicants can trust the listed qualifications.

Implementation Details

  • Link job listings to employer DIDs
  • Implement cryptographic signing of job postings
  • Create verification mechanisms for job authenticity
  • Update UI to display verification status of jobs

Current Status

‚úÖ Implemented

Job listings are now cryptographically linked to employer DIDs, with signature generation and verification implemented. This allows applicants to verify that jobs are legitimately posted by the claimed companies without relying on a trusted job board.

Step 3

Verifiable Credentials Implementation

Start 60% Complete Complete

The third step is to implement W3C Verifiable Credentials for storing and sharing claims about applicants. These credentials are cryptographically verifiable and can be used to prove qualifications without revealing the underlying data.

Implementation Details

  • Add verifiable credential structures to models
  • Implement credential issuance endpoints
  • Create credential verification logic
  • Add credential management UI
  • Integration of DIDKit for Interoperability:
    • Use DIDKit to issue and verify Verifiable Credentials, ensuring compatibility with external systems
    • Support multiple DID methods (did:web, did:key) per W3C standards
    • Leverage DIDKit's support for Ed25519Signature2020 to align with current proof types
    • Prepare for BBS+ signatures to enable selective disclosure in Step 5

Current Status

‚úÖ Implemented

Verifiable Credentials support has been added to the backend using DIDKit for interoperability. The system now supports issuing, storing, and verifying credentials with cryptographic proofs across multiple DID methods (did:web, did:key) and proof types (Ed25519Signature2020), with preparation for BBS+ signatures to enable future selective disclosure capabilities.

Standard Example: ZKP with Trua Trust Bureau

üîí Trua Trust Bureau ZKP Flow:

In our architecture, the ZKP process works as follows:

  1. Applicant authorizes Trua to generate a ZKP for a specific job application
  2. Trua Trust Bureau holds the private data and generates the ZKP
  3. The ZKP proves the applicant meets requirements without revealing actual data
  4. Neither the employer nor the applicant ever sees the underlying private data
  5. Trua acts as the trusted source of truth throughout the process
